# Harness Design: 让 AI 自主编写完整应用的"脚手架"工程学

> **博客**：[Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)
> **作者**：Prithvi Rajasekaran（Anthropic Labs）
> **发表日期**：2026-03-24

---

## 先说几个基础概念

在读这篇博客之前，有三个概念值得先解释清楚：

**什么是 Harness（脚手架）？** 在 AI Agent 的语境下，harness 指的是**包裹在模型外面的一整套编排系统**。模型本身只会生成文本，而 harness 决定了：什么时候让模型开始工作、工作成果交给谁检查、检查不合格怎么处理、上下文太长了怎么办。拿一个你可能熟悉的例子来说：**Claude Code 就是一种 harness**——它在 Claude 模型外面包了一层系统，负责管理对话、调用工具、压缩上下文等等，用户感知到的"Claude Code 很好用"，很大程度上是 harness 的功劳而非模型裸能力。

Harness 里面有很多**组件（零件）**，每个组件解决一个具体问题。比如这篇博客提到的：上下文压缩（compaction）、上下文重置（context reset）、独立评估 Agent（evaluator）、sprint 分解等等——它们都是 harness 的零件。作者的一个核心观点是：**每个零件本质上都是一块"补丁"，用来弥补模型自身的某个缺陷**。比如模型会"上下文焦虑"→ 加一个 context reset 零件来解决。当模型升级后不再焦虑了，这个零件就可以拆掉。所以 harness = 一套补丁的集合，需要随模型进化而增减。

**什么是全栈应用（Full-stack Application）？** 一个完整的软件应用通常分为几层：用户在浏览器里看到和点击的"前端"（界面、按钮、动画），在服务器上处理业务逻辑和数据的"后端"（API、数据库），以及把两者连起来的胶水。"全栈"就是指这些层全都包含在内——不只是一个好看的页面，而是一个真正可以运行、有数据存储、有业务逻辑的完整应用。博客中让 AI 构建的"2D 像素游戏制作器"和"浏览器端音乐工作站"就是全栈应用的例子。

**什么是朴素实现（Naive Implementation）？** 这里指的是最简单、最直接的做法——不做任何特殊设计，直接把任务丢给一个 AI Agent，让它从头写到尾。就像让一个程序员独自坐在电脑前，不给代码审查、不分阶段、不做项目管理，从第 1 行代码一直写到最后一行。这篇博客的出发点就是：**这种朴素做法为什么行不通，以及怎么改进**。

**什么是 Sprint Contract（冲刺合同）？** 来自敏捷开发的概念。"Sprint"是一个短周期的工作阶段（比如"这一轮我们只做关卡编辑器"），"Contract"是在这个阶段开始前，干活的人和验收的人先坐下来约定好的一份清单——**做到什么程度算完成、怎么验证算合格**。就像装修前业主和施工队签的验收标准："墙面平整度误差不超过 3mm、插座高度统一 30cm"。没有这份约定，干完活后双方容易扯皮——"我觉得做完了""我觉得没做完"。在这篇博客里，sprint contract 就是 Generator（写代码的 Agent）和 Evaluator（测试的 Agent）在每轮开工前协商好的验收标准。

## 一句话总结

这篇博客的核心发现是：让 AI 自主编写复杂应用时，**把"写代码"和"检查代码"拆给不同的 AI Agent 做**（借鉴 GAN 的对抗思想），再加上一个负责做项目规划的 Agent，构成 Planner → Generator → Evaluator 三 Agent 架构，就能在数小时的自主会话中产出比单 Agent 好一个量级的结果。同时，作者展示了随着模型能力提升（Opus 4.5 → 4.6），harness 中的某些组件可以被去掉——**harness 的每个组件都是对模型缺陷的补丁，模型变强了补丁就可能过时**。

## 问题：直接让 AI 连续编码会怎样？

### 问题一：上下文焦虑（Context Anxiety）

LLM 有一个"上下文窗口"的限制——可以粗略理解为它的"工作记忆"容量。当 AI Agent 连续编码几个小时，对话历史会越来越长。这时候一个奇怪的现象出现了：**模型开始"紧张"，觉得自己快没记忆空间了，于是匆匆忙忙地收尾**——本来还有很多功能没做完，它就宣布"大功告成"了。这在 Sonnet 4.5 模型上尤为严重。

作者比较了两种应对方法：

- **压缩（Compaction）**：把之前的对话历史做个摘要，腾出空间，但还是同一个 Agent 继续干。问题是——模型虽然"记忆"被压缩了，但它仍然"知道"自己已经聊了很久，焦虑感依旧。
- **重置（Context Reset）**：彻底清空记忆，换一个全新的 Agent 接手。前一个 Agent 把自己的工作成果和进度写成一份"交接文档"，新 Agent 读了这份文档后继续干。代价是：交接过程需要额外设计和成本。

在 Sonnet 4.5 上，压缩不够用——必须用重置。这是早期 harness 设计中的一个核心约束。

### 问题二：自我评价的宽容偏差

另一个大问题是：**AI 对自己写的东西总是过度自信**。当你让它检查自己的代码或设计时，它倾向于说"嗯不错不错，做得很好"——即使一个人类一眼就能看出质量平庸甚至有 bug。

这在"设计好不好看"之类的主观判断上特别明显（因为没有"能不能跑通"这样的客观标准来打脸）。但即使是能客观验证的功能问题，AI 的自我判断力也出奇地差。

作者的关键洞察是：**让一个人既当运动员又当裁判很难调教，但专门训练一个"裁判"让它严格挑剔，却容易得多**。一旦有了外部的严格反馈，"运动员"就有了具体的改进方向。

## 解法一：前端设计的"生成-评价"循环

作者先在前端设计（就是网页的视觉外观）领域做实验，因为自我评价问题在这里最突出——"好不好看"没有客观标准。

### 把"好看"变成可打分的标准

为了让"这个设计好不好看"变成 AI 可以操作的评价，作者定义了四个打分维度：

| 维度 | 在评什么 | 被强调程度 |
|:---|:---|:---:|
| **设计质量** | 颜色、排版、布局是否统一协调，形成独特的风格感？ | 高 |
| **原创性** | 是有自己的设计想法，还是一看就是模板/AI 自动生成的那种千篇一律的样子？ | 高 |
| **工艺** | 间距是否一致、字体大小有没有层次、颜色搭配是否和谐等技术细节 | 低 |
| **功能性** | 抛开美不美，用户能不能顺畅地使用这个界面？ | 低 |

为什么"设计质量"和"原创性"权重高，而"工艺"和"功能性"权重低？因为 **Claude 在技术执行上已经做得不错了**（毕竟 LLM 的强项是遵循规则），但在美感和创意上容易产出"AI 味十足"的平庸设计——那种到处都是紫色渐变和白色卡片的千篇一律。通过加重前两个维度，明确告诉模型"不要偷懒用通用模板"。

### 怎么运转

整个循环基于两个 AI Agent：

1. **Generator（生成者）**：根据用户描述（如"给一个荷兰艺术博物馆做官网"）生成一个网页
2. **Evaluator（评价者）**：不是看代码或截图，而是**真正打开网页，像用户一样到处点**（通过 Playwright 浏览器自动化工具），然后按上面四个维度打分，并写出详细的改进意见
3. 改进意见发回给 Generator，它据此修改设计
4. 关键机制：Generator 每轮结束后要做一个**战略决策**——如果分数在涨，就继续在当前方向上精打细磨；如果路线不对，就彻底换一种设计风格

每次循环跑 5-15 轮，总时长可达 4 小时。

### 有意思的发现

- **标准的文字本身就能引导模型**。作者在标准里写了一句"最好的设计是博物馆级别的"，这句话就已经把模型推向了更有品味的方向——甚至在 Evaluator 给出任何反馈之前，仅凭这套标准，生成质量就已明显优于没有任何引导的基线。
- **改进不是一条直线**。分数大体上升，但作者经常觉得中间某一轮比最终版更好。
- **会出现创造性的飞跃**。最惊艳的例子：为一个荷兰艺术博物馆做网站，前 9 轮一直在迭代一个深色调的传统落地页。第 10 轮，Generator 突然推翻一切，重新设计成一个 3D 空间体验——用 CSS 渲染出棋盘格地板的展厅、自由悬挂在墙上的画作，用门廊而非滚动来导航。这种创造性飞跃在"一次性生成"中从未见过。

## 解法二：构建完整应用的三 Agent 架构

前端设计的实验验证了"生成-评价分离"的有效性。接下来作者把这个思路扩展到构建完整应用。

### 三个角色各司其职

可以把这三个 Agent 想象成一个小型软件团队：

**Planner（产品经理）**：
- 用户只需给 1-4 句话描述想要什么（如"做一个 2D 像素游戏制作器"）
- Planner 将其扩展为一份**完整的产品规划书**：包含哪些功能、长什么样、分几个阶段做
- 故意**只做高层设计、不写技术细节**——因为如果产品经理在技术细节上瞎指挥犯了错，程序员后面会被坑
- 还会主动找机会在产品里加入 AI 功能（比如"用 Claude 帮用户自动生成游戏关卡"）

**Generator（程序员）**：
- 按计划**一个功能一个功能地实现**（每个功能叫一个 sprint，类似敏捷开发的冲刺）
- 每个 sprint 开始前，先和 Evaluator **协商"完成标准"**——就像程序员和测试在开工前先对齐"什么样算做完了"。这份约定叫 sprint contract
- 有版本管理工具（git），可以追踪每次改动

**Evaluator（测试/QA 工程师）**：
- 真正**打开运行中的应用，像用户一样操作**——点按钮、填表单、测功能
- 按 sprint contract 里约定的条目逐条验证，发现问题就详细记录
- 每个维度有及格线，不及格就把 sprint 打回去，连带详细的 bug 报告

Agent 之间怎么沟通？不是聊天，而是**通过写文件、读文件**——一个 Agent 写好一份文件（比如"第 3 个 sprint 的评测报告.md"），另一个 Agent 读取并响应。

### Sprint Contract 为什么重要？

产品规划书是高层级的（"用户可以编辑关卡"），但测试需要具体标准（"用户拖拽矩形工具时应该填充整个区域"）。Sprint contract 就是在两者之间搭桥——在写代码之前，程序员和测试先协商好这一轮具体做什么、怎么验收。这避免了程序员埋头苦干半天，结果测试说"这不是我要的"。

### 效果对比：一个 Agent 单干 vs 三 Agent 协作

用同一句话 "Create a 2D retro game maker" 做对比：

| 维度 | 单 Agent（20分钟, $9） | 三 Agent 协作（6小时, $200） |
|:---|:---|:---|
| 功能范围 | 基础编辑器 + 游戏运行 | 16 个功能、10 个 sprint，含 AI 辅助创作 |
| 界面质量 | 大片空白浪费、操作流程不清晰 | 更精致、布局合理、视觉风格统一 |
| 核心功能 | **游戏根本跑不起来**——角色放上去了但不响应任何操作 | 游戏可以玩（虽然物理引擎有些粗糙） |
| AI 功能 | 无 | 内置 Claude 对话框，可以用自然语言生成游戏的各个部分 |

成本高了约 20 倍，但质量差异是"完全不能用"和"基本可用且功能丰富"的区别——这不是 20% 的改进，而是质的飞跃。

### Evaluator 到底能抓到什么 bug？

来看几个 Evaluator 真实发现的问题：

| 事先约定的验收标准 | Evaluator 发现 |
|:---|:---|
| 矩形填充工具：拖拽一个区域，整个区域都该被填充 | FAIL — 只在拖拽的起点和终点放了方块，中间全空的。填充函数写了但没被正确调用 |
| 用户可以删除已放置的实体出生点 | FAIL — 按删除键需要两个条件同时满足，但点击实体只触发了其中一个 |
| 可以通过接口重新排列动画帧顺序 | FAIL — "重排序"这个接口路径被另一个接口路径截胡了，系统以为你在传一个数字类型的 ID，报错了 |

但让 Evaluator 表现得这么好**不是开箱即用的**。Claude 默认是个糟糕的"测试工程师"——它会发现一个真正的 bug，然后自己说服自己"啊这个应该不是什么大问题"就放过了。也容易只做表面测试，不去探索边界情况。作者花了很多轮去调教：读 Evaluator 的日志 → 找到它判断失误的地方 → 修改它的 prompt → 再来一遍，反复多次才调到满意。

## 模型升级后的简化：从 Opus 4.5 到 Opus 4.6

### 核心理念：harness 的每个零件都是一块"补丁"

作者提出了一个很有启发性的观点：**harness 中的每个组件，本质上都是在为模型的某个缺陷打补丁**。比如：

- Sprint 分解 → 补丁对象：模型无法持续连贯地工作太久
- Context Reset → 补丁对象：模型在长对话中会焦虑
- Evaluator → 补丁对象：模型自我评价不靠谱

当模型升级了，某些缺陷可能已经被修复，对应的补丁就应该被移除。作者一开始尝试一次性砍掉很多组件，但失败了——因为无法判断哪些是还需要的、哪些是可以去掉的。后来改为**一次只去掉一个，观察效果**，这样才看清每个组件到底有没有用。

### Opus 4.6 带来了什么？

Opus 4.6 发布时的描述：

> "规划更仔细、能持续执行更长的 Agent 任务、在大代码库中更可靠、代码审查和调试能力更强"

这些正好对应了 sprint 分解和 context reset 要补的缺陷。结果：

- **Sprint 分解可以去掉了**——Opus 4.6 能连贯工作 2 个多小时不走偏
- **Context Reset 也可以去掉了**——上下文焦虑在 4.6 上基本消失。整个构建过程作为一个连续会话运行，靠 SDK 的自动压缩管理记忆增长

### Evaluator 的角色变了

去掉 sprint 后，Evaluator 从"每做完一小段就检查"变成了"全部做完后检查一次"。

更重要的变化是：**Evaluator 是否有用，取决于任务是否接近模型的能力边界**。

- 在 4.5 上，几乎所有构建任务都在模型的能力边界附近 → Evaluator 全程有用
- 在 4.6 上，模型自身更强了 → 以前需要 Evaluator 才能做好的事情现在模型自己就能搞定 → Evaluator 对这部分来说成了多余的开销
- 但对于仍然接近模型极限的部分（更复杂的功能），Evaluator 仍然有实质贡献

所以 **Evaluator 不是"一律要加"或"一律不加"的决定，而是要看当前模型对当前任务的可靠性**。

### 简化后的 harness 做了什么？

用简化版 harness + Opus 4.6 构建了一个浏览器端的音乐制作软件（DAW，类似简版 GarageBand）：

| 阶段 | 时长 | 成本 |
|:---|:---:|:---:|
| Planner 规划 | 4.7 min | $0.46 |
| 第一轮编码 | 2 hr 7 min | $71.08 |
| 第一轮 QA | 8.8 min | $3.24 |
| 第二轮编码 | 1 hr 2 min | $36.89 |
| 第二轮 QA | 6.8 min | $3.09 |
| 第三轮编码 | 10.9 min | $5.88 |
| 第三轮 QA | 9.6 min | $4.06 |
| **合计** | **3 hr 50 min** | **$124.70** |

最终产出了一个能用的音乐制作程序：有编曲视图、混音器、音频传输控制。作者还完全通过对话让内置的 AI Agent 创作了一小段曲子——Agent 自己设定了调性和节拍，铺上旋律，加入鼓轨，调整混音电平，加上混响效果。

QA Agent 仍然发现了 Generator 遗漏的关键问题：录音按钮只是个摆设（按了什么都不会发生）、音频片段不能拖拽移动、效果器面板只有数字滑块而没有图形化的均衡器曲线。这说明即使模型更强了，在最后一英里的完善上，独立 Evaluator 仍然有价值。

## 思考与讨论

**"写代码的和检查代码的不能是同一个人"——这条人类工程铁律对 AI 同样成立**。这篇博客最核心的洞察可以归结为：LLM 是糟糕的自评者，但可以是优秀的互评者。这跟 GAN 的成功逻辑一致——来自对手的反馈比自我反省更有效。在 agent 系统设计中，这意味着凡是需要质量把关的环节，都应该交给独立 agent 来做，而不是让执行者"自查"。

**"好"的定义越具体，AI 做得越好**。作者发现，仅靠评分标准里的文字描述（还没有任何评估反馈介入之前），生成质量就已经明显提升了。这暗示一种容易被忽视的技巧：与其说"做得好一点"，不如给 AI 一套**明确的评分维度和权重**。"要有创意"是模糊的，"原创性占 30% 权重，抄模板扣分"就是具体的。把主观的"好"拆成多个具体维度，AI 就有了可操作的目标。

**Harness 里的每个零件都是一块补丁——模型变强了补丁就该拆**。正如开头解释的，harness 由很多组件构成，每个组件弥补模型的一个缺陷。Sprint 分解弥补的是"模型干久了会走偏"，context reset 弥补的是"模型干久了会焦虑"。当 Opus 4.6 让这两个缺陷基本消失后，这两个零件就多余了。所以每次模型升级，都应该回头审视："这个零件补的缺陷还在吗？"作者总结得好——harness 的设计空间不会随模型进步而缩小，而是**移动**。过度复杂的 harness 不仅浪费钱，还可能因为限制了模型本来能自己处理好的事情而反而降低效果。

**动手之前先对齐标准——人类的项目管理经验直接适用于 AI**。Generator 和 Evaluator 在写代码之前先协商"做到什么程度算完"（sprint contract），本质上就是人类软件团队里的"验收标准评审"。这避免了程序员干完活被测试以不同标准打回的低效循环。人类工程实践中那些解决协调问题的方法（sprint 规划、代码审查、设计评审），很多可以直接搬进多 Agent 架构。

**成本翻倍换来的是质的飞跃，不是量的改善**。单 Agent 花 $9 得到一个核心功能坏掉的应用；三 Agent 花 $200 得到一个功能丰富且基本可用的应用。这不是"贵一点好一点"的线性关系，而是"不能用 vs 能用"的分水岭。当然，对于简单任务（比如做一个静态页面），三 Agent 架构就过度设计了。什么复杂度的任务配什么复杂度的 harness，是一个需要判断的问题。

**让 Evaluator 真正"用"应用，而不只是看代码**。Evaluator 通过 Playwright（浏览器自动化工具）真正打开应用、点按钮、填表单——就像一个真人用户在试用。这比看静态代码或截图能发现更多问题，因为很多 bug 只在真正运行和交互时才会暴露（比如"这个按钮按了没反应"这种问题，光看代码你可能觉得写对了）。代价是每轮评估需要真实时间，但发现问题的质量远高于静态分析。

---

## 关键 Takeaways

1. **写和查必须分开**：AI 自我评价不靠谱，独立的 Evaluator Agent + 明确的评分标准才能形成有效的质量反馈
2. **把"好"拆成具体维度**：与其说"做好一点"，不如给出带权重的多维评分标准——这是一种被低估的 prompt 技巧
3. **动手前先对齐标准**：Agent 之间在写代码前协商"什么叫完成"，避免做无用功
4. **Harness 的每个零件都是补丁**：每个组件弥补模型的某个缺陷（如 context reset 补"焦虑"、sprint 分解补"干久走偏"），模型升级后要逐个审视哪些补丁可以拆掉
5. **真正操作 > 静态检查**：让 Evaluator 像用户一样打开应用操作，能发现只在运行时暴露的 bug
6. **从简开始，按需增加**：不要一上来就设计最复杂的架构，而是先观察失败模式，再针对性地加组件
