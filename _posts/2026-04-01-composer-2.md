> **论文**：[Composer 2 Technical Report](https://arxiv.org/abs/2603.24477)
> **作者**：Cursor Research Team
> **机构**：Cursor (Anysphere)

Cursor 的编程 agent 模型 Composer 经历了三代迭代，这篇技术报告描述了最新的 Composer 2 是如何训练出来的。核心路径很直接：拿一个通用大模型（Kimi K2.5，1.04T 参数 / 32B 活跃参数的 MoE 模型），先在代码数据上做持续预训练，再用大规模强化学习在真实编程环境中训练 agent 行为。在他们自建的 CursorBench 上，Composer 2 达到 61.3% 准确率，比上一代 Composer 1.5 提升了 37%，与 GPT-5.4、Opus 4.6 等前沿模型处于同一水平，而推理成本更低。

*Figure 1: Composer 2 在 CursorBench 上的表现，与前代及其他前沿模型的对比*

## 持续预训练：给 RL 打好地基

Composer 2 的训练分两个阶段，第一阶段是持续预训练（continued pretraining）。在此之前需要选定基座模型。团队评估了 GLM-5、Kimi K2.5、DeepSeek V3.2 三个开源模型，评估维度有三个：

1. **Coding knowledge**：用内部 benchmark FreshBench 测试。FreshBench 的构造方式是从之前 Composer 模型的使用记录中，找到模型需要阅读库源码或搜索网页才能完成的 case，从中提取问答对，再用搜索 agent 验证答案的正确性。这些问题对应的是模型"不知道"的编码知识。
2. **State tracking**：类似于公开的 LoCoDiff benchmark，要求模型在经历多次 diff 后还原文件状态，衡量长期记忆能力。评估指标用字符级编辑距离而非精确匹配，因为精确匹配对单字符错误过于敏感。
3. **Codebase perplexity**：在 Cursor 私有 monorepo 上计算负对数似然，衡量模型对真实代码的建模能力。

论文还评估了多个闭源模型作为参照，完整结果如下（表中 FreshBench 越高越好，State Tracking 和 Neg. Log-Likelihood 越低越好）：

| 模型 | FreshBench ↑ | State Tracking ↓ | Neg. Log-Likelihood ↓ |
|:---|:---:|:---:|:---:|
| DeepSeek V3.2 | 68.9% | 66 | 11.75M |
| Kimi K2.5 | 83.2% | 86 | 13.81M |
| GLM-5 | 79.2% | 92 | 14.11M |
| GPT-5.4 | 92.5% | 103 | - |
| Claude 4.6 Opus | 88.9% | 65 | - |
| Gemini 3 Flash | 84.5% | 27 | - |
| Claude 4.5 Sonnet | 80.1% | 69 | - |
| Claude 4.5 Haiku | 61.7% | 177 | - |

三个开源候选中，Kimi K2.5 在 FreshBench 上领先（83.2%），State Tracking 居中（86），perplexity 也居中（13.81M）。最终选了 Kimi K2.5（1.04T 参数 / 32B 活跃参数的 MoE 模型），除了综合表现好之外还考虑了推理基础设施的效率。

选型策略上有两点值得注意。第一，团队故意不用 coding agent benchmark 来选基座模型，因为他们发现 agent 能力在 RL 阶段会发生剧烈变化，基座模型在 agent benchmark 上的表现对最终效果的预测性不强。换句话说，选基座模型时更应该看"知识储备"和"基础能力"，而非"当前的 agent 表现"。第二，这里存在一个方法论上的张力：论文称这是"base model selection"，但实际评估的模型（Kimi K2.5、GLM-5、DeepSeek V3.2）发布时都已经过 SFT 甚至 RLHF，并非纯粹的 base model。FreshBench 和 State Tracking 测到的能力中有多少来自 post-training、多少来自模型底层知识，难以分离。比如 DeepSeek V3.2 的 FreshBench 最低（68.9%）但 perplexity 也最低（11.75M），这两个维度给出了相反的信号，最终哪个更能预测持续预训练 + RL 后的效果，论文没有进一步消融。当然，大多数开源 MoE 模型目前不公开纯 base checkpoint，这可能也是现实约束。

选定基座后，在大量代码数据上做持续预训练，目的是进一步提升编码知识，为后续 RL 阶段提供更好的起点。持续预训练本身分三个子阶段：主体在 32k 上下文长度上训练，然后做 256k 长上下文扩展，最后在特定编码任务上做一轮短 SFT。训练使用 MXFP8 精度，在 NVIDIA B300 上完成。

一个关键的实验验证是：持续预训练的质量确实能传导到下游 RL 效果。团队在较小的 Qwen3-Coder-30B-A3B 上做了控制实验，用三个对数等间距的 token 量（即不同的计算预算）分别做持续预训练，产出三个 checkpoint，然后对每个 checkpoint 分别做 SFT + 相同的 RL 训练。结果表明，SFT 后的 cross-entropy loss 越低，RL 训练后的 reward 越高（Figure 2 左），两者之间存在清晰的正相关关系。

![Figure 2](/assets/images/composer-2/pretraining_rl_curves_3.png)
*Figure 2: 左图是 Qwen3-Coder 上的控制实验，展示预训练 loss 与下游 RL reward 的正相关关系；右图是 Kimi K2.5（Composer 2 的基座模型）持续预训练过程中 perplexity 的对数线性下降*

此外，为了加速生产环境的推理，团队额外训练了 Multi-Token Prediction (MTP) 层用于投机解码（speculative decoding）。MTP 层从零初始化，使用自蒸馏方式训练——让 MTP 层去预测主 LM head 在每个位置的 logit 分布。为了保证泛化性，MTP 层的训练起点是持续预训练中期的 checkpoint，而非最终版本。

## 强化学习：在真实编程环境中学习

### 训练设置

RL 阶段是 Composer 2 训练的核心。训练任务的构成（Figure 3）覆盖了软件工程的多种场景，包括 bug 修复、代码生成、重构等，远比公开编程 benchmark 丰富。在训练后期，团队根据 rollout 的 turn 数和 thinking token 数等启发式指标来上采样更难的数据点。

![Figure 3](/assets/images/composer-2/x1.png)
*Figure 3: RL 训练任务的类别分布*

训练环境尽可能模拟真实的 Cursor 使用场景——使用与生产环境相同的工具集和 harness，agent 可以读写文件、运行 shell 命令、搜索代码库、搜索网页等。

算法层面，Composer 2 使用带有多样本分组的 policy gradient 方法，每个 prompt 只训练一次（single-epoch regime）。相对于 GRPO 的几个关键修改：

1. **去掉长度标准化**：GRPO 原始公式中的长度标准化项会引入长度偏差，参照 Dr. GRPO 的发现将其移除。
2. **不做组内 advantage 标准化**：当一个组内所有 rollout 的正确性相同时，标准差归一化会将微小的行为差异放大，导致退化。
3. **不做超长 rollout 遮掩**：有些工作建议 mask 掉超出最大序列长度的 rollout，但团队在小规模实验中未观察到收益，因此不采用。

### KL 散度估计器的选择

一个技术细节值得关注。RL 训练中使用 KL 散度 $\mathrm{KL}(q\|p) = \mathbb{E}_{x\sim q}[-\log r(x)]$（其中 $r(x) = p(x)/q(x)$）作为正则化。很多开源 RL 实现使用 $k_3 = (r-1) - \log r$ 这个估计器，它在 $p$ 和 $q$ 接近时是低方差的无偏估计。但当两个分布偏离较大时，$k_3$ 的方差会急剧增大（Figure 4）。Composer 2 改用最基础的 $k_1 = -\log r$ 估计器。

![Figure 4](/assets/images/composer-2/kl_estimator_variance.png)
*Figure 4: 不同 KL 估计器在两个高斯分布上的方差比较，$k_3$ 在大 KL 值时方差爆炸*

### 异步 RL 与稳定性

由于 agent rollout 可以非常长（数百次 tool call），系统在高度异步的环境下运行。稳定性的关键策略是最小化 off-policy 程度：

- 推理 worker 支持 rollout 中途更新权重（mid-rollout weight update），使后续 token 更接近当前策略
- 训练端使用 MoE routing replay——推理时记录每层选择的 expert 索引，训练时重放这些路由决定，避免因数值差异导致的 expert 分配不一致
- 权重同步采用 delta 压缩，每步只传输差量，1T 参数模型的更新压缩到几个 GB

### 一个令人鼓舞的发现

近期文献中有不少工作认为 LLM 上的 RL 主要是将概率质量集中到已知的好轨迹上，可能以牺牲策略多样性为代价。但 Composer 2 的训练过程中，average 性能和 best-of-K 性能同时提升（Figure 5），没有出现此消彼长的 trade-off。这暗示 RL 不仅仅是在重新加权已有的推理路径，而是在扩展模型能到达的正确解的集合。

### Self-Summarization 与长程连贯性

为了让 agent 在长期任务中保持连贯性，Composer 2 沿用了 Composer 1.5 的 self-summarization 技术。每次 rollout 可以包含多个 generation，中间由模型自行生成的摘要连接。RL 的 reward 信号作用于整条链中模型产出的所有 token——好的轨迹中的摘要会被上调权重，丢失关键信息的摘要则被下调。

### 非线性长度惩罚

为了让模型在简单任务上快速完成、复杂任务上多思考，训练中加入了一个凹增的非线性长度惩罚：

$$
C_{\text{length}\{k,q\}}(x) = \frac{(1+kx)^{1-q} - 1}{k(1-q)}
$$

其中 $k$ 和 $q$ 控制曲线的弯曲程度，$x$ 是 thinking token 数、tool call token 数、工具输出 token 数等的加权组合。这个函数的性质是：在 $x$ 较小时（简单任务），每增加一点 effort 的惩罚更陡；在 $x$ 较大时（复杂任务），惩罚增长趋缓。效果之一是模型学会了并行发起多个 tool call 以提高效率。

## CursorBench：为什么需要自建评估

团队认为公开 benchmark（如 SWE-bench）与真实编程任务之间存在四个关键差距：

1. **领域不匹配**：SWE-bench 集中在 bug 修复，Terminal-Bench 包含不少抽象谜题而非典型工程任务
2. **Prompt 过度指定**：公开 benchmark 的 prompt 通常非常详细，而真实开发者请求往往模糊且允许多种方案
3. **数据污染**：OpenAI 曾暂停报告 SWE-bench Verified 结果，因为发现前沿模型能从记忆中生成 gold patch
4. **评估维度单一**：只看功能正确性，忽略代码质量、延迟、成本等

CursorBench 从 Cursor 工程团队的真实编程会话中提取任务。与公开 benchmark 相比，CursorBench 任务需要修改的代码多一个数量级（中位数 181 行 vs SWE-bench 的 7-10 行），而 prompt 更短、更模糊（中位数 390 字符 vs 1185-3055 字符）。

CursorBench 会持续迭代更新。从第一版到第三版，任务涉及的文件数和代码行数翻了一倍多，任务类型也从单纯的代码编辑扩展到了命令执行监控、数据分析等场景。

## 基础设施亮点

论文的基础设施部分篇幅很大，这里提炼几个有意思的点：

**训练并行策略**：Composer 2 使用 Context Parallelism (CP) 替代 Tensor Parallelism (TP) 作为主要的长上下文扩展轴，因为 CP 通信开销更小。EP 与 TP 解耦，使得 expert parallel 度可以独立调整。在持续预训练阶段使用 EP=8, CP=2，RL 阶段使用 EP=8, CP=8。

**低精度训练的细节**：MoE 前向传播使用了一种改良的 NVFP4 格式——将原始的 per-tensor scaling 改为 per-token scaling。原因有两个：per-tensor scaling 使训练变成 batch-variant 的，导致 RL 训练发散；而且 inter-token 的 scale 值会泄露未来 token 的信息。反向传播则使用标准 MXFP8 格式，因为反向只在训练集群上运行，不需要与推理端一致。

**环境系统 Anyrun**：训练中的代码执行环境跑在 Cursor 内部的 Anyrun 平台上——每个环境是一个独立的 Firecracker VM，支持 fork 和 snapshot。单集群每秒可调度 500+ pods。训练横跨 3 个 GPU 区域和 4 个 CPU 区域。

|:---|:---:|:---:|:---:|
| Composer 2 | 61.3 | 73.7 | 61.7 |
| Composer 1.5 | 44.2 | 65.9 | 47.9 |

几个关键观察：
- Composer 2 相比基座模型 Kimi K2.5 在 CursorBench 上从 36.0 提升到 61.3，提升了 70%
- 在公开 benchmark 上，Composer 2 与 GPT-5.4、Opus 4.6 等处于同一水平
- 在推理成本上，Composer 2 处于 Pareto 最优前沿——准确率接近最强模型，成本接近小模型

## 讨论

**专精模型 vs 通用模型**的路径选择是这篇报告传递的核心信息。Composer 2 证明了一条可行的路径：拿一个强通用模型做 domain-specific 的持续预训练 + RL，可以在特定领域达到甚至超越远大于自身的通用前沿模型，同时推理成本更低。这对于所有有明确垂直场景的 AI 产品来说都是一个有参考价值的策略。

**RL 训练中 average 和 best-of-K 同时提升**这一发现与最近文献中"RL 主要是 redistribution 而非 discovery"的观点形成了对比。一个自然的问题是：这种差异是来自 agent 任务本身的特性（action space 更大、exploration 更丰富），还是来自他们的特定训练设置（如单 epoch、无 advantage 标准化等）？如果能在更多设置下复现这一现象，对理解 RL post-training 的本质会很有帮助。

**CursorBench 的设计思路**——从真实用户会话中提取任务、持续迭代更新——比大多数静态 benchmark 更接近真实场景。但它是内部非公开的，其他团队无法复现这些数字。论文中也坦诚地指出了 Haiku 4.5 在 SWE-bench Verified 上接近 GPT-5 这种"公开 benchmark 区分度不足"的现象，但这也意味着 CursorBench 本身的结果同样需要外部验证。
