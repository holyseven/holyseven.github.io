# Inside OpenAI's In-House Data Agent：六层上下文让数据 Agent 从能用到可靠

> **博客**：[Inside OpenAI's in-house data agent](https://openai.com/index/inside-our-in-house-data-agent/)
> **作者**：Bonnie Xu, Aravind Suresh, Emma Tang（OpenAI）
> **发表日期**：2026-01-29

---

OpenAI 把自己的数据平台（3500+ 用户、600PB、70000 个数据集）包了一层 AI Agent，让员工可以用自然语言完成从找表、写 SQL、跑分析到出报告的完整流程。底层模型是 GPT-5.2，构建工具包括 Codex、Evals API 和 Embeddings API——全是 OpenAI 自家的开发者产品。

博客最有价值的部分不是"我们做了个 Agent"，而是**为什么让它可靠这么难，以及需要哪些工程决策**。核心答案是一套六层上下文体系：从表结构元数据到代码级语义理解，每一层解决一类特定的"Agent 会犯的错"。此外，博客还展示了交互设计、评估机制、安全模型和三条值得跨行业借鉴的实战教训。

---

## 问题：找对表比写 SQL 更难

70000 张表听起来夸张，但在 OpenAI 这个规模的公司里是正常的。博客提到使用者横跨 Engineering、Product、Research、Go-To-Market、Finance、Data Science 等部门，数据仓库中典型会包含：产品行为数据（ChatGPT / API 的用户日志、会话统计、留存漏斗）、业务与财务数据（订阅收入、API 计费、客户合同）、工程指标（推理延迟、错误率、服务可用性）、实验数据（A/B 测试结果和分组）、模型训练元数据（评测结果、版本对比）等。之所以膨胀到 70000 张，是因为同一份原始数据会被按不同粒度聚合（日/周/月）、不同维度切片（按地区/按产品线/按用户类型）、不同团队各自建视图，再加上历史快照和废弃但未清理的旧表。

在这个规模下，数据分析的瓶颈不是"不会写 SQL"，而是两件更前置的事。

**第一，找到正确的表本身就是个挑战。** 博客引用了一位内部用户的原话："我们有很多长得很像的表……有的包含未登录用户，有的不包含；有的字段重叠，很难分清哪个是哪个。"在企业数据仓库中，表的命名和 schema 趋同但语义分裂是常态——两张表的列名可能完全一致，但一张统计的是第一方流量，另一张包含全部流量。

**第二，写出正确的 SQL 充满隐蔽陷阱。** 多对多 join 让行数悄悄膨胀、过滤条件放错位置导致结果失真、NULL 值在聚合中被吞掉。博客展示了一条 180+ 行的真实查询——光人工审查就很痛苦，更不用说从零写对。

分析师的时间应该花在"定义指标、验证假设、做决策"上，而不是"找表、调 SQL、排查 bug"上。

## 核心设计：六层上下文体系

**Agent 的回答质量完全取决于它拿到的上下文质量。** 博客明确指出：没有上下文，即使 GPT-5 也会严重高估用户数或误解内部术语。为此，Agent 围绕六层递进的上下文构建。

![六层上下文体系：从表使用元数据（最底层、最易获取）到运行时上下文（最顶层、按需触发），信息深度递增](/assets/images/openai-data-agent/layers-of-context.svg)

### Layer 1: 表使用元数据

最基础的一层。包含两类信息：

- **Schema 元数据**：列名、数据类型、表的血缘关系（上下游依赖）——Agent 据此构造 SQL 的基本骨架
- **历史查询模式**：过去的分析师怎么用这张表、通常和谁 join——Agent 据此学习"正确的用法"而非只看结构

### Layer 2: 人工标注

业务专家为表和列撰写的描述，捕捉业务含义、设计意图和已知陷阱。这些信息无法从 schema 自动推断——"这张表只统计付费用户""这个字段 2024 年前后含义变了"这类知识只存在于人脑和文档中。

博客在这里做了一个关键过渡："元数据不够。要真正区分两张表，你需要理解它们是怎么被创建的、数据从哪里来。"

### Layer 3: Codex 代码级理解

![Codex 代码分析流水线：从数据管道代码中提取表的语义定义](/assets/images/openai-data-agent/codex-enriched-pipeline.svg)

**这是整个体系中最有特色的一层。** 数据仓库里的表不是人手动填的，而是由自动化程序（pipeline）定时生成的——比如一段 Spark 作业每天读取原始用户行为日志，过滤掉机器人流量，按用户 ID 聚合，计算会话数和消息数，写入 `daily_user_activity` 表。**这些 Spark / Python / SQL 程序就是"数据管道代码"**，它们精确定义了每张表的数据从哪来、经过了什么过滤和转换。Agent 通过 Codex 分析这些管道代码，从而获得一个代码级的表定义。

这层提供了前两层无法给出的信息：字段值的唯一性约束、数据更新频率、数据的粒度和范围（比如是否排除了特定事件类型）、以及表在 SQL 之外的系统中的使用方式。

一个直观的例子：两张表的 schema 几乎一模一样，但看了 pipeline 代码后，Agent 就能判断出一张只包含 ChatGPT 第一方流量，另一张包含所有流量。这个区别在 schema 和人工标注中都不一定出现。

而且这层上下文会自动刷新——代码变了理解就跟着更新，不需要人工维护文档。

### Layer 4: 组织知识

Agent 可以检索 Slack 消息、Google Docs 和 Notion 文档。这些非结构化内容包含大量 Agent 必须知道的背景：

- 产品上线时间线（"这个功能上周二才上线，之前查不到数据"）
- 可靠性事件记录（"3 月 5 日数据缺失是管道故障"）
- 内部代号和工具名称的真实含义
- 关键指标的官方计算逻辑和口径定义

这些文档被提取、嵌入向量、并**带上权限元数据**存储。运行时通过一个带权限控制和缓存的检索服务拉取——不是所有人都能看到所有文档。

### Layer 5: 记忆

用户纠正了 Agent 的错误后，Agent 可以保存这个纠正，下次遇到同类问题时自动应用。

博客给了一个具体案例：Agent 不知道怎么按某个 A/B 实验过滤数据——需要匹配一个特定字符串（实验的 gate 定义），不能靠模糊匹配。被纠正一次后就永远记住了。

记忆分两个层级：**全局记忆**（所有人共享，避免每个人重复纠正同一个问题）和**个人记忆**（个人偏好和工作习惯）。用户可以手动创建和编辑记忆。

博客用一个直观的对比展示了记忆的价值——同一个问题"ChatGPT Image Gen 过去 30 天的登录 DAU 是多少？"，没有记忆时 Agent 花了 22 分 41 秒（因为找不到正确的表），有了记忆后只需 1 分 22 秒：

| 无记忆 | 有记忆 |
|:---:|:---:|
| ![无记忆：耗时 22 分 41 秒](/assets/images/openai-data-agent/without-memory.png) | ![有记忆：耗时 1 分 22 秒](/assets/images/openai-data-agent/with-memory.png) |

### Layer 6: 运行时上下文

当前五层都不够用时——比如遇到一张全新的、从未被标注过的表——Agent 直接向数据仓库发出实时查询，亲自检查表里有什么数据。它还能与 Airflow、Spark 等数据平台系统对话，获取更广泛的运行时上下文。

### 整体检索架构

![上下文检索架构：离线流水线汇总 + Embeddings 向量化 + RAG 运行时检索](/assets/images/openai-data-agent/context-retrieval.svg)

每天一个离线流水线将前几层信息（表使用、人工标注、Codex 代码分析）汇总为统一的归一化表示，通过 Embeddings API 转为向量并存储。查询时通过 RAG 只拉取最相关的上下文片段喂给模型，即使面对 70000 张表，运行时延迟也可预测且可控。运行时查询（Layer 6）则按需直接访问数据仓库。

## 交互设计：像队友而不是工具

Agent 不是"输入问题 → 输出答案"的一次性接口，而是被设计成一个可以来回对话的协作者。

下图是博客中的一个示例：用户问"NYC 出租车哪些路线最不靠谱"，Agent 自主完成了从规划分析方案、搜索表、检查 schema、写 SQL、到汇总结论的完整流程：

![Agent 分析 NYC 出租车数据的完整推理过程：规划 → 搜索表 → 检查 schema → 写查询 → 汇总](/assets/images/openai-data-agent/nyc-taxi-example.png)

**对话式推进**：Agent 跨轮次保持完整上下文。用户可以直接追问"那按客户类型拆呢？"而不需要重述背景。如果 Agent 走偏了，用户可以中途打断并重新引导——就像和一个会听话的人类同事协作。当用户的意图不清时，Agent 主动反问确认；如果没回应，就用合理默认值（比如默认看最近 7 或 30 天）继续推进，不阻塞流程。

**闭环自我纠错**：Agent 不是按固定脚本跑的。它会检查每一步的中间结果——如果查询返回 0 行（可能是 join 或 filter 写错了），它会自行排查原因、调整方案、重试，整个过程不需要用户介入。博客特别强调这种"将迭代从用户转移到 Agent 自身"的设计理念。

**Workflow 模板**：上线后发现用户频繁重复跑同样的分析（每周业务报告、表数据验证等），于是支持将常用分析封装成可复用的指令集，编码一次 best practice 后全团队共享。

**多入口部署**：Slack bot、Web 界面、IDE、Codex CLI（MCP 接入）、内部 ChatGPT（MCP 连接器）——去用户在的地方，而不是让用户来找你。

## 评估：像单元测试一样持续守护质量

一个持续演进的 Agent，质量可能悄悄退化。OpenAI 基于 Evals API 构建了一套自动化评估体系。

**什么是 Evals API？** 它是 OpenAI 提供的自动化评测服务，解决的问题是：你做了一个基于 LLM 的应用，怎么系统性地持续检查它的输出质量？核心流程是：定义一组"问题 + 预期正确答案"的测试集 → 把问题发给你的 AI 应用拿到实际输出 → 用一个 grader（通常也是 LLM）对比预期和实际，自动评分。它和人工评审的区别在于可以持续自动运行——每次改了 prompt、换了模型、更新了上下文，都能自动跑一遍全部测试集看有没有退化。本质上就是给 AI 应用做持续集成（CI）。

评估的基本单元是一组**问答对**：每个问题对应一条人工编写的"黄金 SQL"（golden SQL），代表已验证的正确查询。评估时，将自然语言问题发给 Agent，执行 Agent 生成的 SQL，然后与黄金 SQL 的执行结果对比。

关键设计是**不做 SQL 字符串匹配**。语法不同的 SQL 完全可以产出相同的正确结果，结果集也可能包含额外的列但不影响答案的正确性。因此评估流程是：将 Agent 的 SQL + 执行结果和黄金 SQL + 执行结果一起送给 grader 模型，由 grader 综合判定正确性并给出评分和解释。

![评估流水线：自然语言问题 → Agent 生成 SQL → 执行并与黄金 SQL 结果对比 → grader 评分](/assets/images/openai-data-agent/eval-pipeline.svg)

这些 eval 的角色类似持续运行的单元测试——开发阶段用来**抓退化**（regression，即本来能正确回答的问题因为改了 prompt 或换了模型反而答错了，质量"倒退"），上线后做**金丝雀监控**（canary monitoring，来自"矿井里的金丝雀"——矿工带金丝雀下井，金丝雀对毒气更敏感，它先倒下说明空气有问题；这里指持续用一小组测试问题探测 Agent 质量，一旦这组"金丝雀问题"的表现下降，就是系统出问题的早期预警）。

## 安全模型：纯透传，不增加权限

Agent 的安全设计原则很明确：**它只是一个界面层，不创造任何新的访问权限**。

- 用户只能查到自己本来就有权访问的表
- 权限不够时 Agent 会提示，或退回到用户有权限的替代数据集
- 全过程可审计——Agent 展示推理步骤和假设，并链接到底层查询结果供用户直接检查原始数据

这个"pass-through"原则在 Layer 4（组织知识检索）上尤为关键——Slack、Google Docs、Notion 中的文档在被嵌入存储时就带上了权限元数据，运行时检索服务强制执行权限校验。

## 三条实战经验

**Less is More。** 早期把全套工具暴露给 Agent，工具之间存在功能重叠。人类能凭经验区分何时用哪个工具，但 Agent 会困惑。解法：收窄和合并工具调用，减少歧义。

**Guide the Goal, Not the Path。** 高度详细的 prompt 指令反而降低了结果质量。分析问题的"大框架"相似但细节差异很大，过于刻板的指令把 Agent 推向错误路径。解法：只给高层目标（"分析用户增长趋势"），让 GPT-5 自己选择执行路径。当基座模型足够强时，过度编排反而有害。

**Meaning Lives in Code。** Schema 描述表的"形状"，查询历史描述表的"用法"，但表的**真正含义藏在生成它的代码里**。Pipeline 逻辑中的假设条件、新鲜度保证、业务意图——这些在 SQL 或元数据里从不出现。通过 Codex 爬取代码库，Agent 理解的是数据集"实际上是怎么被构造出来的"，而不仅仅是"表面上长什么样"。

## 思考与讨论

**六层上下文是一个有参考价值的框架，但更值得注意的是它背后的设计逻辑**。这六层不是随意堆砌的——它们按照"获取成本递增、信息深度递增"排列，形成一个上下文金字塔。Layer 1-2（元数据 + 人工标注）是传统 text-to-SQL 方案已有的；Layer 3（代码语义）和 Layer 4（组织知识）是大多数同类方案完全缺失的；Layer 5-6（记忆 + 运行时查询）则是 Agent 自身的学习和兜底机制。一个值得追问的问题是：如果你只能投入资源建设其中三层，应该优先建哪三层？博客的隐含答案似乎是 Layer 1 + 3 + 5——schema 提供骨架，代码提供语义，记忆提供持续学习能力。

**Codex 代码分析这一层的信息密度可能被低估了**。传统方案把"数据字典"理解为人工维护的文档，但在实际企业环境中，数据字典的维护质量参差不齐、更新严重滞后。代码是**活的数据字典**——它精确定义了每张表的构造逻辑，并且随系统演化自动更新。OpenAI 选择用 Codex（代码理解工具）而非通用文档检索来攻克这一层，是一个有意思的架构选择：它把"理解数据"的问题转化为"理解代码"的问题，而后者恰好是 LLM 最擅长的。

**记忆系统的全局/个人分层设计解决了一个实际问题**。没有记忆的 Agent 每次对话都从零开始——上次告诉它"这个实验要按特定字符串过滤"，下次它又忘了。全局记忆避免 N 个人重复纠正同一个错误，个人记忆尊重个体的工作习惯和偏好。这比简单的"对话历史"更有结构——它把纠正固化为可复用的知识，而不是淹没在对话流中。

**"少即是多"和"指导目标不指导路径"——这正在成为跨公司的共识**。上一篇 [Harness Design](2026-03-28-harness-design-long-running-apps.md) 中 Anthropic 得出的核心教训也是"从最简方案出发、只在需要时增加复杂度"，并且发现 harness 的每个组件都是对模型缺陷的补丁，模型变强了补丁就可能过时。OpenAI 在这里独立地发现了同样的规律：给 Agent 更多工具和更详细的指令并不是好事，会增加歧义、限制模型推理的灵活性。两家公司都在说同一件事——**当基座模型足够强时，最好的编排是最少的编排**。

**评估体系的"不做字符串匹配"是一个容易被忽视但很关键的设计**。如果评估只比较 SQL 文本，大量语法不同但结果正确的查询会被误判为失败，团队会被虚假的"质量下降"警报淹没。用 grader 模型同时看 SQL 和执行结果来判定正确性，本质上是把评估问题也交给了 AI——这本身也引入了新的可靠性问题（grader 也可能犯错），但相比字符串匹配是一个显著的改进。

**权限透传在 Layer 4 上的挑战值得单独关注**。Agent 检索 Slack、Google Docs、Notion 内容时，每条文档必须带上权限元数据，检索服务必须在运行时校验权限。否则 Agent 可能把 A 部门的内部讨论泄露给 B 部门——这在企业场景中是不可接受的安全事故。博客只用了一句话带过（"stored with metadata and permissions"），但实际实现中这可能是工程量最大、最容易出 bug 的部分。

**博客没有给出的定量数据**。3500 用户、70000 数据集、600PB——这些规模数字说明系统在真实运行。但文中没有给出 Agent 的准确率（SQL 正确的比例）、用户满意度、使用频率或对分析师效率的量化影响。作为工程博客可以理解，但如果想评估这套方案在其他企业的可复制性，这些数据会很关键。
