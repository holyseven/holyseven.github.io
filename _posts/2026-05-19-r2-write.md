## 问题：深度推理在写作任务上为什么不好使？

Deep reasoning（长链思维推理）在数学等可验证领域带来了巨大提升——DeepSeek-R1 在 AIME25 上相比 V3.1 提升 99.3%。但在开放式写作任务上，情况截然不同。论文首先给出了一组数据：

| 基准 | Qwen3-30B-A3B (No Think → Think) | Qwen3-8B (No Think → Think) | DeepSeek V3.1 → R1 |
|------|:-:|:-:|:-:|
| WritingBench | +1.2% | +0.8% | +0.3% |
| HelloBench | +0.7% | +0.4% | +0.7% |
| MATH 500 | +10.5% | +11.1% | +7.2% |
| AIME 25 | +200.4% | +221.5% | +99.3% |

写作任务的提升幅度不到 2%，与数学领域两个数量级的差距形成鲜明对比。

## 原因分析：写作推理中缺少 verification 和 backtracking

论文用 Claude-4.5-Sonnet 对推理轨迹做模式标注，将思维模式分为五类：Subgoal Setting、Backtracking、Verification、Correction、Backward Chaining。

![Thinking pattern analysis](/assets/images/r2-write/x1.png)
*Figure 1: 思维模式分布对比。上排为模式分布，下排为各模式对最终答案质量的贡献比例。*

关键发现：在数学任务中，Verification 和 Backtracking 是获得正确答案的主要驱动力；而在写作任务中，模型几乎只做 Subgoal Setting（规划大纲），缺乏对草稿的反思和回溯修改。

这个观察引出了论文的核心假设：开放式写作中的质量缺陷（需求遗漏、事实错误、逻辑矛盾、重复等）是可以通过 reflection 和 revision 来识别和修正的，类似于数学推理中的 error correction。

## R2-Write 方法：Writer-Judge 协作合成 + Process Reward RL

![Pipeline overview](/assets/images/r2-write/x2.png)
*Figure 2: R2-Write 流程总览，包含 query 选择、数据构造和 RL 三个阶段。*

### 数据选择：找到模型最需要反思的地方

给定 query $q$，计算参考答案分数 $S^q_r$ 与模型答案分数 $S^q_m$ 的差距 $\Delta_q = S^q_r - S^q_m$，选择 gap 最大的 top-k 样本。最终选出 3K 样本做 SFT、5K 样本做 RL。

数据来源包含两类：创意写作（来自 DeepWriting 数据集的 20K 实例，覆盖 14 个类别）和报告生成（10K 深度研究 query，覆盖 10 个类别）。

### SFT 数据合成：Writer-Judge 迭代

数据构造的目标有两个：(1) 在思维轨迹中注入 reflection 和 revision 模式，同时保留已有的规划模式；(2) 确保每次反思是有效触发的——只在草稿确实存在问题时才反思，且修改后质量有提升。

具体流程：
1. Writer 模型生成初始答案
2. Judge 模型基于评估 rubric 给出分数和反馈
3. Writer 将 Judge 反馈内化为自己的思考：先 reflection（识别问题），再 revision（提出修改方案）
4. Writer 生成修订后的答案
5. 仅保留分数提升的修订轮次

关键细节：Step 3 中刻意注入自然的自我反思风格，比如 "Hmm...maybe I should revise..." 或 "Wait, I found that..."，避免机械化模式。迭代终止条件为累积分数超过阈值或达到最大轮次。

### 强化学习：Process Reward 防止冗余反思

SFT 之后模型获得基本的 reflection/revision 能力，但可能过度触发（无意义的反思会浪费 token）。RL 阶段的核心是引入 process reward 来约束反思质量。

总体 reward 设计：

$$R_{\text{all}} = \begin{cases} \alpha R_a + (1-\alpha) R_p, & \text{if } R_a > 0 \\ R_a, & \text{otherwise} \end{cases}$$

其中 $R_a$ 是答案级别 reward（pairwise comparison with reference），$R_p$ 是过程级别 reward。只有当 $R_a > 0$（即最终答案质量可接受）时才计算 $R_p$，防止模型通过构造漂亮的推理过程来 hack reward 而答案本身很差。

Answer Reward 采用 pairwise comparison：让 LLM Judge 比较模型输出 $\mathbf{x}$ 与高质量参考 ref，赢得 1、平 0.5、输 0。

Process Reward 的计算更精细。首先从思维轨迹中提取所有 reflection 段落 $\mathcal{M} = \{M_{F_1}, \ldots, M_{F_K}\}$，每个段落从三个维度评判：

- Issue Identification $R_{\text{find}}^{(i)} \in \{-1, +1\}$：识别的问题是否真实存在
- Revision Suggestion $R_{\text{rev}}^{(i)} \in \{-1, +1\}$：修改建议是否符合 rubric
- Execution Alignment $R_{\text{align}}^{(i)} \in \{-1, +1\}$：最终答案是否执行了修改计划

单个 reflection 段的奖励：

$$R_p^{(i)} = \begin{cases} +1, & \text{if } R_{\text{find}}^{(i)} > 0 \text{ and } R_{\text{rev}}^{(i)} > 0 \text{ and } R_{\text{align}}^{(i)} > 0 \\ -1, & \text{otherwise} \end{cases}$$

总 process reward 为所有段落的均值：$R_p = \frac{1}{K} \sum_{i=1}^K R_p^{(i)}$

这个设计很直接：只有三个条件同时满足（找到了真问题、给出了合理修改方案、最终确实执行了修改），这次反思才能获得正奖励。任何一个条件不满足就给负奖励。

## 实验结果

论文在 WritingBench、DeepResearch-Gym、DiscoX 三个基准上做了全面评测。

### 主要对比

| 方法 | WritingBench Score | DeepResearch-Gym KPR | DiscoX Acc. |
|------|:-:|:-:|:-:|
| Qwen3-8B (base) | 71.84 | 63.78 | 18.80 |
| Qwen3-8B + Distill (SFT) | 78.82 | 66.37 | 21.70 |
| Qwen3-8B + R2-Write-SFT | 80.71 | 70.05 | 22.70 |
| Qwen3-8B + R2-Write-SFT + RL | 82.22 | 71.63 | 23.16 |
| Qwen3-8B + R2-Write-SFT + RL_p | 83.80 | 72.50 | 23.50 |
| GPT-5 | 85.87 | 76.26 | 42.62 |
| Gemini2.5-Pro | 82.15 | 73.20 | 40.23 |

8B 参数的模型在 WritingBench 上达到 83.80，超过了 Gemini2.5-Pro（82.15），接近 GPT-5（85.87）。

### Reflection 模式修正了哪些问题

论文分析了 R2-Write 相比 baseline 获胜的 case，将反思修正的问题归为三类：

| 数据集 | 需求对齐 (RA) | 事实/逻辑修正 (FLC) | 质量提升 (QE) |
|------|:-:|:-:|:-:|
| WritingBench | 10.50% | 19.88% | 69.62% |
| DeepResearch-Gym | 24.17% | 34.72% | 41.11% |

在创意写作中，质量提升（完整性、流畅度、深度）占比最大；而在研究报告生成中，事实/逻辑修正的比例明显更高——这对应了模型需要核对材料中的信息是否在报告中被正确引用的场景。

### 对通用能力的影响

| 方法 | AIME25 | MMLU-Pro |
|------|:-:|:-:|
| Qwen3-8B + Distill | 66.78 | 59.62 |
| R2-Write-SFT | 67.84 | 60.42 |
| R2-Write-SFT + RL_p | 64.78 | 59.22 |

数学推理和通用知识能力基本保持不变，说明写作领域的 reflection/revision 训练没有损害模型的其他能力。

### RL 算法和 Reward 设计的消融

| 方法 | WritingBench Score |
|------|:-:|
| Qwen3-8B (no RL) | 71.84 |
| Pointwise + PPO | 73.42 |
| Pointwise + GRPO | 75.20 |
| Pairwise + GRPO | 77.87 |
| Pairwise + PPO | 78.42 |

Pairwise reward 优于 pointwise reward（对开放式写作这种没有标准答案的任务，比较式评估更能区分质量差异）；PPO 优于 GRPO。

## 思考与讨论

1. 训练样本的拼接结构值得审视。论文 Algorithm 1 显示最终的训练样本结构为：`<think> 初始规划 T_0 + 初稿 a_q^(0) + 反思修订段 R_q^(1) + ... </think> <answer> 最终答案 a_q^* </answer>`。初稿被整体放进了 `<think>` 块，只有最终修订版出现在 `<answer>` 中。这个设计的动机是让整个 draft-reflect-revise 循环能在单次推理轨迹内完成，从而适配标准 RL（PPO）的优化框架——如果初稿作为 response 输出，reflection 就变成多轮对话，credit assignment 会复杂得多。但代价也很明显：写作的初稿本身可能几千字，塞进 `<think>` 后思考 token 急剧膨胀（Figure 3 显示 thinking 轨迹动辄上万 token）；而且 SFT 构造阶段的初稿原本是 writer 模型正式生成的 response，事后回收到 `<think>` 里属于人工拼接，推理时模型需要在 thinking 中"假装"写一篇完整文章再反思，这和数学推理中在 thinking 里做推导的认知模式有本质区别。这更像是一个工程妥协——为了让循环塞进单次推理框架。

2. Process Reward 的设计取舍。三个条件的"全部满足才给正奖励"是一个严格的 AND 逻辑。如果模型识别了真实问题、给出了合理方案但最终没执行（可能因为上下文长度限制），就会被惩罚。这种设计倾向于减少反思次数（宁可不反思也不做无效反思），从 token 效率角度看是合理的。但在极端情况下可能导致模型过于保守，放弃一些本可以修正的问题。

3. 8B 模型 vs 闭源大模型的差距。R2-Write 在 WritingBench 上已接近 GPT-5，但在 DiscoX（专家级翻译）上差距仍然很大（23.50 vs 42.62）。这可能反映了 reflection/revision 能弥补的是"知道应该怎么做但初稿没做好"的情况，而对于需要更多世界知识和领域专业性的任务，模型本身的能力上限仍是瓶颈。
