# 论文读书笔记生成

将学术论文（通常是机器学习/AI领域）转化为面向有一定基础但非专家读者的中文读书笔记。

读者画像：没有读过论文原文，了解基本概念，有一定数学基础但不多，不是理论专家。笔记需要提供足够的上下文让这类读者能独立理解内容。

## 输入处理

用户会提供以下内容之一：
- 论文的 arXiv 链接
- 论文 PDF 文件路径（通过 Read 工具读取）
- 直接粘贴的论文文本

从中提取标题、作者、机构等基本信息，并基于论文全文生成笔记。

### 获取论文内容和图片

原始论文 HTML 和文本缓存在 `.paper-cache/{论文简称}/` 目录（已加入 .gitignore，不会被提交）。图片最终保存到 `assets/images/{论文简称}/`（会被提交和部署）。

1. 如果论文来自 arXiv，**第一步用 `curl` 将完整 HTML 页面下载到本地文件**（如 `.paper-cache/{论文简称}/paper_raw.html`）。后续所有操作——文本提取、图片 URL 解析、figcaption 查找——都从这个本地文件进行，不再重复请求 arXiv。
2. **不要使用 WebFetch 工具访问 arxiv.org**，该工具对 arxiv 域名有限制会报错。直接用 Bash 的 `curl` 命令。
3. 从本地 HTML 中提取论文正文文本，保存为纯文本文件备用（如 `.paper-cache/{论文简称}/paper_full_text.txt`），作为后续写笔记的参考源。
4. 从本地 HTML 中提取图片路径和 `<figcaption>` 信息，确认每张图片对应论文中的哪个 Figure。图片完整 URL 通常为 `https://arxiv.org/html/{arxiv_id}/figures/xxx.png` 或 `https://arxiv.org/html/{arxiv_id}/x{n}.png`，用 `curl` 下载到 `assets/images/{论文简称}/` 目录。
5. 在笔记中用绝对路径引用图片：`![描述](/assets/images/{论文简称}/x1.png)`

## 笔记内容要素

以下是笔记**可能包含**的内容要素。不要把它当作固定模板逐条填充——根据每篇论文的特点，灵活选择哪些要素值得展开、哪些可以合并或省略、以什么顺序组织最自然。

### 必要要素

- **论文元信息块**：正文开头（front matter 之后、标题之后）用 blockquote 格式列出论文基本信息，论文标题带超链接。格式如下：
  ```
  > **论文**：[论文英文标题](论文链接)
  > **作者**：完整作者列表
  > **机构**：作者所属机构
  ```
- **快速概览**：让读者在几秒内判断这篇论文是否值得深入了解。形式不限——可以是一句话总结，可以是一个对比（"之前...，这篇论文发现..."），可以是一个问题引入。
- **核心方法/发现**：这是笔记的主体。对关键公式：先给出公式，逐项解释符号，给出直觉解读。善用类比。算法用自然语言描述而非粘贴伪代码。
- **实验证据**：提炼关键数字，指出最有说服力和最薄弱的结果。用 Markdown 表格呈现关键数据（从论文表格中提取，不要截图）。消融实验只取最有信息量的 2-3 个发现。
- **思考与讨论**：以"读者视角的好奇心"而非"审稿人视角的挑刺"来写。重点是帮助读者理解方法的适用边界和未来可能的延伸方向。语气上用"值得关注的是..."、"一个自然的问题是..."、"如果进一步..."这类探讨式表述，而非"论文未能..."、"严重限制了..."这类否定式表述。记住这是公开 blog，作者可能会看到，要保持尊重和善意。可参考的讨论角度：适用场景的边界、工程落地的挑战、可能的改进方向、与其他方法的互补关系等。挑 2-3 个最有价值的点展开即可。

### 可选要素（根据论文特点决定）

- **背景铺垫**：如果论文依赖的前置知识不太常见，用类比和直觉解释铺垫。如果读者大概率已知，可以极简带过或省略。
- **理论分析**：如果论文有理论部分，解释定理的直觉含义和实际意义，指出简化假设与现实的差距。没有理论部分就不写。
- **与相关工作的对比**：如果论文方法与某个已有方法形成有趣的对比或递进关系，值得展开。
- **定性案例**：如果论文提供了有说服力的 case study 或可视化，可以引用并分析。

## 结构组织原则

**核心原则：让内容决定结构，而非结构决定内容。**

- 不同类型的论文应有不同的笔记风格：
  - 方法论文：以"问题 → 直觉 → 形式化 → 实验验证"为主线
  - 实证发现类论文：以"现象 → 假设 → 验证 → 解释"为主线
  - 系统/工程类论文：以"设计目标 → 架构选择 → 工程权衡 → 评估"为主线
  - 分析类论文：以"观察 → 度量方法 → 发现 → 启示"为主线
- 标题和子标题应反映论文本身的逻辑，不要用"第一部分"、"第二部分"这样的通用标题
- 各部分篇幅比例应随论文特点灵活调整，不强制规定百分比
- 如果论文有递进关系的多个方法版本（如先 baseline 再改进），按递进关系展开并解释为什么需要改进

## 图表使用原则

- **关键结果图**必须包含：论文的核心对比图、主实验结果表格
- **方法示意图**优先包含：帮助理解方法流程的架构图/示意图
- **图文必须对应**：引用图表时，必须具体解释图中的坐标轴、标注点、关键趋势各代表什么，让读者只看笔记文字就能和图对上，而不是泛泛地说"如图所示"
- 数据表格用 Markdown 表格重制，提取最关键的行列，不需要完整复制
- 每张图/表都应有简短的中文说明，解释读者应该从中看到什么
- 消融实验、超参数分析等图表视信息量决定是否包含

## 写作风格

1. **语言**：中文为主，术语保留英文原文。学术但不晦涩，允许口语化表达辅助理解。
2. **公式格式**：行内公式用 `$...$`，独立公式用 `$$...$$`（前后换行）。每个关键公式后必须跟自然语言解释。
3. **类比适度**：可以用类比辅助理解抽象概念，但不要过度使用比喻。避免拟人化表述（如"模型脑子里"）和过度修辞。优先用直白的技术语言描述。
4. **客观立场**：不评价论文"好不好"，直接分析内容本身。对缺乏充分实验支持的声明保持合理怀疑，指出实验设计的局限（如样本量不足、验证范围有限等）。
5. **避免千篇一律**：每篇笔记的开头、过渡、总结方式都应有变化。不要每篇都用相同的引入句式或段落结构。
6. **篇幅控制**：总长度 2000-4000 字，视论文复杂度灵活调整。
7. **术语和概念要交代清楚**：论文中出现的关键术语、缩写、指代（如实验中的变量名、分组标签）必须在首次出现时明确解释其含义，不要让读者去猜。
8. **完整性**：论文提到了 N 个并列的要点/维度/方法，就应该全部介绍，不要只展开部分而省略其余。如果某些确实不重要，应明确说明省略理由。
9. **避免 AI 腔**：不要用"精巧的设计""巧妙地""值得注意的是"等空洞的评价性修饰语，直接陈述事实和逻辑。减少加粗强调的使用频率。

## 输出文件

### 文件命名（Jekyll 兼容）

笔记文件放在 `_posts/` 目录，文件名必须带日期前缀，格式为：

```
_posts/YYYY-MM-DD-{论文简称}.md
```

- 日期使用笔记生成当天的日期（通过系统获取）
- 论文简称用小写英文、单词间用 `-` 连接，不加 `_notes` 后缀
- 示例：`_posts/2026-02-27-think-deep-not-just-long.md`

### Front Matter

每篇笔记文件开头必须包含 Jekyll YAML front matter，格式如下：

```yaml
---
layout: post
title: "论文中文标题或简述"
date: YYYY-MM-DD
categories: [paper-notes]
tags: [相关标签, 如 reasoning, LLM, test-time-compute]
paper_title: "论文原始英文标题"
paper_authors: "第一作者 et al."
paper_link: "arXiv 链接或 DOI"
---
```

- `title`：固定前缀 `【论文笔记】`，后接中文简明概括，如 `"【论文笔记】用 Deep-Thinking Tokens 度量 LLM 推理努力"`
- `tags`：3-5 个与论文内容相关的关键词标签
- `paper_link`：如果有 arXiv ID 则填 `https://arxiv.org/abs/{id}`

### 图片与目录

- 每篇论文的图片保存到 `assets/images/{论文简称}/`，其中论文简称与笔记文件名一致（去掉日期前缀）
  - 示例：`assets/images/bytescale/x7.png`，对应笔记文件 `_posts/2026-03-04-bytescale.md`
- 在笔记中用绝对路径引用：
  ```markdown
  ![Figure 7](/assets/images/bytescale/x7.png)
  *Figure 7: ByteScale 系统总览*
  ```
- 如果目录不存在则先创建
- 图片格式必须为 PNG/JPG（Markdown 无法渲染 PDF）。如果从 LaTeX 源码获取的图片是 PDF 格式，用 `sips -s format png input.pdf --out output.png` 转换
